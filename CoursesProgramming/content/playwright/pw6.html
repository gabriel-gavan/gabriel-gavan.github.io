Locator Chaining and Filtering

Chaining narrows a search progressively; filters match nested text or elements. This avoids brittle long selectors and keeps intent clear.

Chaining to the actionable element

await page
  .locator('.card')
  .filter({ hasText: 'Pro Plan' })
  .getByRole('button', { name: 'Choose' })
  .click();

Filter by nested elements

const rows = page.getByRole('row');
const activeRows = rows.filter({ has: page.getByRole('cell', { name: /Active/ }) });
await expect(activeRows).toHaveCount(5);

// Filter by text in any descendant
const cheapRows = rows.filter({ hasText: /\$?([0-4]?\d(\.\d{2})?|50\.00)/ });
await expect(cheapRows.first()).toBeVisible();

nth, first, last, and count

// Pick a specific card
await page.locator('.card').nth(2).getByRole('button', { name: 'Choose' }).click();

// Assert exactly 3 notifications are visible
await expect(page.getByRole('alert')).toHaveCount(3);

// Interact with the last row's menu
await page.getByRole('row').last().getByRole('button', { name: 'Menu' }).click();

Best practices

  • Write helpers that return a locator (not a resolved element) to preserve auto‑waiting.
  • Keep chains short; combine semantic locators with minimal CSS when needed.
  • Co‑locate intent‑revealing helpers near tests or in small page objects/components.

Practice

  1. Write planCard(name) that returns the locator for a pricing card; use it to click Choose.
  2. Filter a table to rows with a price under $50 using has/hasText and a regex.
  3. Add an assertion that the chosen plan name appears in the checkout summary.
Tip: Returning locators from helpers keeps Playwright’s retries and auto‑waiting intact. Avoid await locator.elementHandle() unless absolutely necessary.
