Assertions Deep Dive

Choose the right assertions to avoid flakiness. Prefer user‑visible outcomes and leverage auto‑waiting built into expect.

Common assertions

await expect(page).toHaveURL(/\/dashboard/);
await expect(page).toHaveTitle(/Acme/);
await expect(page.getByRole('button', { name: 'Submit' })).toBeEnabled();
await expect(page.getByTestId('total')).toHaveText('$19.00');
await expect(page.getByRole('img', { name: 'Avatar' })).toBeVisible();

Soft vs. hard assertions

// Hard: fails the test immediately on error
await expect(page.getByRole('heading')).toBeVisible();

// Soft: collects failures and continues (report all at the end)
await expect.soft(page.getByText('Promo')).toBeVisible();

Timeouts, polling, and stability

// Override timeout for slow operations
await expect(locator).toHaveText('Done', { timeout: 10_000 });

// Use toHaveScreenshot() for visual checks (stabilize fonts/animations first)
await expect(page).toHaveScreenshot('dashboard.png');

Advanced: toHaveJSProperty, toContainText, regex

await expect(page.locator('#price')).toContainText(/\$\d+\.\d{2}/);
await expect(page.locator('video')).toHaveJSProperty('paused', true);
await expect(page.getByRole('alert')).toHaveText(/(Saved|Updated) successfully/);

Best practices

  • Assert on outcomes users see (text, URL, aria states) rather than implementation details (classes).
  • Place assertions soon after the action they validate to shorten traces.
  • Use soft assertions to validate multiple UI elements in a final state view.

Exercises

  1. Replace a textContent() poll with expect(...).toHaveText.
  2. Add three soft assertions that validate a dashboard loaded state (balance, plan tier, avatar visible).
  3. Write a regex assertion for a price format in $12.34.
Tip: Prefer locating by role + name, then asserting state (visible, enabled, checked) or content (text). This reads like a user story and resists markup churn.
