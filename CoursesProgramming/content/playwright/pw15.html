Network Mocking and Interception

Stabilize tests by mocking requests and simulating failures. Interception lets you control server responses, latency, and errors to verify UI behavior.

Intercept and fulfill

// Fulfill a request with static JSON
await page.route('**/api/products', route => {
  route.fulfill({ status: 200, contentType: 'application/json', body: JSON.stringify([{ id: 1, name: 'Pro' }]) });
});

Continue and modify

// Pass through with modified headers
await page.route('**/api/*', route => {
  const headers = { ...route.request().headers(), 'x-test': '1' };
  route.continue({ headers });
});

Simulate latency and errors

// Slow response to test loading states
await page.route('**/api/report', async route => {
  await new Promise(r => setTimeout(r, 1200));
  await route.fulfill({ status: 200, body: JSON.stringify({ ok: true }) });
});

// 500 error to test error UI
await page.route('**/api/checkout', route => route.fulfill({ status: 500, body: 'Server error' }));

Context‑level routing

// Applies to all pages in the context
const context = await browser.newContext();
await context.route('**/feature-flags', route => route.fulfill({ body: '{"beta":true}' }));

Best practices

  • Prefer API seeding over intercepting core flows so E2E still exercises real backend logic.
  • Use interception for edge cases and failure paths that are hard to trigger reliably.
  • Keep mocked data in small fixtures to reuse across tests.

Exercises

  1. Mock a slow network response and assert a loading state appears.
  2. Simulate a 500 error and verify an error banner is shown.
  3. Apply a context‑level route for feature flags and verify a feature toggle in the UI.
Tip: Reset routes between tests to avoid leaks: await page.unroute('**/api/*') or create a fresh context.
