CSS Selectors and XPath

Use CSS/XPath when semantics are missing (e.g., custom widgets). Keep selectors short, stable, and scoped through parent locators to reduce flakiness.

CSS patterns that age well

// Scope then select with a short class
const card = page.locator('.plan-card').filter({ hasText: 'Pro' });
await card.locator('.buy-button').click();

// Attribute selectors are explicit and readable
await page.locator('[data-testid="cart-icon"]').click();
await page.locator('button[data-state="open"]').click();

// :has() can target by descendant presence (supported by Playwright's engine)
await page.locator('.row:has(.status-badge:has-text("Active")) .menu').click();

XPath when relationships are complex

// Row with a specific cell value
await page.locator("//tr[td[contains(., 'Invoice #42')]]//a[normalize-space(.)='View']").click();

// Select sibling by relative position
await page.locator("//label[.='Email']/following-sibling::input").fill('user@example.com');

Guidelines

  • Prefer data-testid for complex, custom components lacking roles/labels.
  • Scope with a parent locator, then use a short selector inside.
  • Avoid index‑based selectors (e.g., :nth-child(5)) unless the order is truly stable.
  • Prefer CSS over XPath when either works; CSS is often shorter and faster to read.

Practice

  1. Add data-testid attributes to a custom dropdown, and update tests to target them.
  2. Refactor one long descendant chain into scoped locator + short selector.
  3. Use :has() to click the context menu of the row that contains a target user.
Tip: Keep selectors intentional. Ask: “If this UI changes slightly, will the selector still point at the same concept?”
