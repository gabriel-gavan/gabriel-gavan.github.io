Fixtures and Test Hooks

Fixtures share setup/teardown across tests; hooks manage per‑test lifecycle. Typed fixtures yield faster, cleaner, and more consistent suites.

Hooks: beforeEach/afterEach

import { test } from '@playwright/test';

test.beforeEach(async ({ page }) => {
  await page.goto('/');
});

test.afterEach(async ({}, testInfo) => {
  if (testInfo.status !== testInfo.expectedStatus) {
    // e.g., emit extra debug logs
  }
});

Per‑test fixture (page already provided)

import { test as base } from '@playwright/test';

type Fixtures = { authedPage: import('@playwright/test').Page };

export const test = base.extend({
  authedPage: async ({ browser }, use) => {
    const ctx = await browser.newContext({ storageState: 'state.json' });
    const page = await ctx.newPage();
    await use(page);
    await ctx.close();
  },
});

export const expect = test.expect;

Worker‑scoped fixture (expensive setup once per worker)

type WFixtures = { seed: { userId: string } };

export const testW = base.extend({
  seed: [async ({ request }, use) => {
    const res = await request.post('/api/seed');
    const body = await res.json();
    await use(body); // available to all tests in this worker
  }, { scope: 'worker' }]
});

Best practices

  • Prefer fixtures that return locators/pages/clients over raw data to keep auto‑waiting.
  • Keep fixtures small and focused; compose them in tests instead of a giant "kitchen sink" fixture.
  • Use worker‑scoped fixtures for costly seeding; clean per‑test artifacts in afterEach.

Exercises

  2. Create an authedPage fixture that reuses state.json and opens the dashboard.
Tip: Prefer reading environment variables inside fixture code, not scattered in tests. Centralizing config reduces surprises.
