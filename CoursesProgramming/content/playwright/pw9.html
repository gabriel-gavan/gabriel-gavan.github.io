Page Object Model (POM)

Encapsulate selectors and flows in classes to keep tests intention‑revealing. Keep POMs small, composable, and focused on user actions, not business assertions.

Simple example

export class LoginPage {
  /** @param {import('@playwright/test').Page} page */
  constructor(page){ this.page = page; }
  get email(){ return this.page.getByLabel('Email'); }
  get pass(){ return this.page.getByLabel('Password'); }
  get submit(){ return this.page.getByRole('button', { name: 'Log in' }); }
  async login(e,p){
    await this.email.fill(e);
    await this.pass.fill(p);
    await this.submit.click();
    await this.page.waitForURL('**/dashboard');
  }
}

Component objects (fragments)

Model reusable widgets (e.g., header, menu, modal) as component objects to avoid massive POMs.

export class Header {
  constructor(root){ this.root = root; }
  get avatar(){ return this.root.getByRole('img', { name: 'Avatar' }); }
  async openProfile(){ await this.root.getByRole('link', { name: 'Profile' }).click(); }
}

Composition over inheritance

  • Avoid deep POM inheritance trees; compose with small objects instead.
  • Return locators from getters and expose high‑level actions as async methods.
  • Keep assertions in tests to separate concerns (POM = actions/selectors).

Exercises

  1. Create a PlansPage with select(name) using chaining + filters.
  2. Extract a Header component that opens the profile menu and clicks Sign out.
  3. Keep assertions in tests; make POM methods return only after navigation stabilizes.
Tip: If a page has many independent areas, split it into fragments (Header, Sidebar, Cart) and compose them in tests.
