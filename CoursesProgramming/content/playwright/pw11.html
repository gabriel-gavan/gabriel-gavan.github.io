Running in CI (GitHub Actions)

Execute tests headless in CI, cache dependencies, shard across workers, and publish artifacts for fast feedback.

Basic workflow

name: e2e
on: [push, pull_request]

env:
  PLAYWRIGHT_BROWSERS_PATH: ~/.cache/ms-playwright

jobs:
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - name: Cache Playwright browsers
        uses: actions/cache@v4
        with:
          path: ~/.cache/ms-playwright
          key: ${{ runner.os }}-browsers-${{ hashFiles('**/package-lock.json') }}
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test --reporter=line,html
      - uses: actions/upload-artifact@v4
        if: always()
        with: { name: report, path: playwright-report }

Matrix across browsers

strategy:
  fail-fast: false
  matrix:
    project: [chromium, firefox, webkit]

steps:
  - run: npx playwright test --project=${{ matrix.project }}

Sharding and retries

# Split the suite across 2 CI jobs
npx playwright test --shard=1/2
npx playwright test --shard=2/2

# Configure retries in playwright.config.ts for CI
retries: process.env.CI ? 2 : 0

Artifacts: traces, screenshots, videos

// playwright.config.ts
export default defineConfig({ use: { trace: 'on-first-retry', screenshot: 'only-on-failure', video: 'retain-on-failure' } });

# Upload test-results and trace zips on failure
- uses: actions/upload-artifact@v4
  if: failure()
  with:
    name: traces
    path: test-results/**/*.zip

Exercises

  1. Add a browser matrix and upload reports for each job (unique artifact names).
  2. Enable sharding to split the suite across two jobs; confirm runtime decreases.
  3. Attach trace and video artifacts on failure for easier triage.
Tip: Keep CI deterministic: pin Node and browser versions, set timezone/locale, and mock non‑deterministic data for visual tests.
